The First Website

Trials, tribulations and lots and lots of googling.

Building Blocks

Building this website has been the best assignment of this class, not only because of how it forced us to pull in skills we have been whetting for the last ten weeks, but because we could express ourselves through a new medium. Man shares such a special faculty with God: the ability to create. Expression is both a "high" and "low" form of life, and learning web development is another weapon in our arsenal of communication.

Despite being a platform for “higher forms” of life, it was a definite challenge. Making it look appealing, planning the layouts, deciding which frameworks to use, and simply pushing through the tough challenges all bestowed skills not present in a traditional “worksheet” or “homework assignment.”

What the Heck to Use?

I’m subscribed to the /r/webdev subreddit, which posts links about HTML, CSS, Javascript and web development in general. It is an excellent resource to browse when attempting to navigate the murky waters of the web development sea. Git and Github are two great tools to manager version control and code from the cloud. I used Bootstrap to style the CSS and make sure all of the columns were nice and dynamic. I used a combination of plain Javascript and a fundamental library called jQuery, which is shorthand for common DOM manipulations. I really enjoyed using jQuery, especially in combination for creating the cards for my article teasers.

Most Difficult Moment

Creating the cards for my articles in a dynamic was the portion of the website I fiddled with the most. I took inspiration from a tutorial on W3 School, which teaches readers how to create a simple card with a drop shadow that increases after an onhover event fires.

My approach was to generalize those cards across the website, so I could reuse the same CSS styles and have a coherent, intuitive card theme. Readers who understand the behavior of one card will implicitly understand future behaviors other cards, even if the content within them is disparate. This is a simple User Experience (UX) principle I attempt to use for future content on the site. The only two examples I have right now are: the card and the links. I don’t actually have the exact same CSS for both, so the onhover styles are slightly different, but in future iterations of the website I’ll change it. Here is a sample of the CSS for the cards:

              
                .article-card {
                  color: var(--color-gray-dark);
                  /* Add shadows to create the "card" effect */
                  box-shadow: 0 4px 8px 0 rgba(0,0,0,0.2);
                  transition: 0.3s;
                  background-color: white;
                }

                .article-card:hover {
                  box-shadow: 0 8px 16px 0 rgba(0,0,0,0.2);
                }

                /* Padding for the card's content */
                .article-card .card-content {
                  padding: 10px 20px 40px 20px;
                }

                .article-card img {
                  object-fit: cover;
                  overflow: hidden;
                }

                .card-content h1 {
                  margin-bottom: 10px;
                  font-size: 24px;
                }
                .card-content a:hover {
                  text-decoration: none;
                }

I like this card theme. I think it take a nod at Google's Material Design. It was a challenge to manage the onhover events for the different cards, but I'll manage in the future to make it more coherent.

Planning for the Future

I deliberately spend most of my time learning the building blocks of the different frameworks and libraries that I need to know to build nice-looking websites. Sure, for the project as a whole I have a measley three pages, but I have cashed in the initial time investment learning the fundamentals of some pretty prominent and popular libraries.

My most worthwhile endeavor was dynamically adding article teaser cards from Javascript onto the page using object oriented programming paradigms (ArticleTeaser class) as well as jQuery, which makes it a pleasure to query and create html elements on the DOM.

I don't have to hand-craft each article page, or each teaser because I can just create a few Javascript objects and have a nice format to send to clients accessing my website.

Final Thoughts

This class was honestly a little too basic for me, and the cirriculum felt like it stifled me from learning at the rate I need to in order to secure a good job or attain a wide skill-set. It's not necessarily the fault of the class itself, I just wish there existed a course for "advanced beginners," meaning people who have experience coding, but not necessarily in web development.

The most learning I accomplished was in the last module about the Google Maps API, which got our feet wet in a world we need to be submerged in. That experience was second only to building the website, which actually was less "taught" and more "self-taught," in the sense that I learned Bootstrap, jQuery, git, Github and all those other frameworks entirely on my own, even though they are industry standards.

I also would have liked to learn an actual framework like React, Angular or Ember, which I hear about all the time in professional web developer talks. I feel like I am ready to explore Javascript through the professional lense, which focuses on solving pragmatic problems to make the developer life easier, rather than simply the possibility of displaying something on the page.
